Beginner Guide

OKX AI Skill Challenge
从大到小讲清楚这个项目

这不是一个“单策略脚本仓库”，而是一套带比赛包装的量化交易系统。它同时包含三条路线：标准化的多组合交易引擎、以 elite_flow 为代表的微观结构自定义策略，以及以 yolo_momentum / yolo_orchestrator 为代表的高杠杆实验路线。目前最稳定可运行的是 elite_flow + dashboard。

入口：engine/main.py 主运行路径：competition demo-start --strategy elite_flow 监控：engine/dashboard.py 状态文件：engine/logs/summary.json
先用一句话理解全局： CLI 负责启动，TradingEngine 负责调度，TradingAlgorithm 负责把“市场数据”变成“目标仓位”，Broker 负责下单，Portfolio 负责记账，StructuredLogger 负责把一切写给 dashboard。

Project Overview

项目总览：它到底是在做什么

这个项目的目标不是只回测，也不是只做前端看板，而是把“量化策略研发、比赛配置、交易执行、运行监控”串成一个完整闭环。你可以把它拆成 4 个层次来理解。

第 1 层
启动层
用 engine/main.py 管理命令、策略列表、比赛模式和守护进程。
第 2 层
决策层
把行情变成信号，再把信号变成目标仓位，再过风险过滤。
第 3 层
交易层
通过 OKX CLI 下单，内部用 Portfolio 单独记账，不直接等于交易所余额。
第 4 层
观测层
把运行状态写到 summary.json、JSONL、CSV，再由 dashboard 读取展示。
通用主线

标准化引擎路线

适合组合策略、回测、定时调仓。核心是 TradingEngine + TradingAlgorithm，路径比较工程化，也更容易扩展。

比赛主线

自定义策略路线

适合像 elite_flow、yolo_momentum 这种需要自己维护事件循环、盘口流、状态机的比赛策略。它们绕开通用调仓引擎，直接自己交易。

为什么要两条线

同一仓库，兼顾复用与速度

通用引擎适合做体系化量化框架；自定义策略适合快速比赛迭代。这个仓库的设计本质上是在平衡“框架一致性”和“比赛执行力”。

Architecture

整体架构图

下面这张图把仓库分成三条运行路径。左边是标准化引擎，中间是当前最关键的比赛路径 elite_flow，右边是 YOLO 系列的实验与编排路径。

路径 A 的价值

如果你想新增一个规范策略，最好接入 core/alpha.py 和 strategies/，这样自动享受组合构建、风控、执行和日志体系。

路径 B 的价值

适合盘口驱动、状态机驱动、比赛速度优先

如果策略依赖订单簿微观结构、分钟级状态切换、主动管理杠杆和持有期，自定义执行循环会更直接。elite_flow 就是这样。

Modules

从大到小拆模块：是什么、为什么、怎么做

1. 命令入口

engine/main.py

是什么：整个项目的总入口。

为什么：统一管理启动、停止、状态查询、比赛命令，避免每个脚本各自为政。

怎么做：解析命令后，分流到标准守护进程或自定义比赛策略；如果是 competition demo-start，会先查注册表，再决定走通用引擎还是自定义策略模块。

2. 策略注册表

competition/registry.py

是什么：比赛策略的配置中心。

为什么：比赛里需要统一记录资金、标的、profile、备注和自定义参数，方便回测、比较、demo 运行共用。

怎么做：读取 engine/config/competition_strategies.json，把 JSON 定义转换成引擎配置或自定义策略配置。

3. 调度引擎

engine/trading_engine.py

是什么：长期运行的交易守护进程。

为什么：需要自动调仓、自动做风险检查、自动保存状态，不依赖 cron。

怎么做：启动后恢复历史状态、启动 WebSocket 价格缓存、按 profile 周期做 rebalance，每 60 秒做 risk check。

4. 核心算法管线

core/algorithm.py

是什么：一套无状态调仓流程。

为什么：把“信号”“组合”“风控”“执行”拆开，方便替换和测试。

怎么做：固定按 Alpha → PortfolioConstruction → Risk → Execution 执行，并统一写日志。

5. 数据层

data/fetcher.py 与 data/feed_ws.py

是什么：历史数据与实时价格输入。

为什么：策略需要历史 OHLCV，风控需要轻量实时价格，比赛路径还要盘口与成交流。

怎么做：历史数据优先走 ccxt / OKX，缓存到 Parquet；实时价格用 ccxt.pro 维护内存缓存，减少 REST 调用。

6. 策略层

strategies/ 与 competition/strategies/

是什么：一部分是标准组合 sleeve，一部分是比赛专用独立策略。

为什么：组合框架和盘口策略的表达方式不一样，不能强行塞在一起。

怎么做：标准策略输出目标权重矩阵；比赛策略则自己维护状态、持仓和下单节奏。

7. 账本层

portfolio/portfolio.py

是什么：内部资金与持仓账本。

为什么：一个 OKX 账户里可能跑多个组合，不能直接拿交易所余额当单个策略净值。

怎么做：内部记录 cash、positions、realized_pnl_total、fees，并能序列化到 engine_state.json。

8. 执行层

execution/broker.py

是什么：交易所桥接层。

为什么：比赛要求订单通过 OKX Agent Trade Kit CLI 打标签，同时行情读取仍然希望保持 ccxt 的便利。

怎么做：ccxt 只做只读查询，真实下单通过 okx swap place，paper 模式返回 synthetic fill。

9. 日志与可视化

logging_/structured_logger.py 与 dashboard/

是什么：状态落盘和展示层。

为什么：查看状态不应依赖 Python 内部对象，日志文件更稳定，也更适合 dashboard。

怎么做：写 summary.json、每组合 JSONL、signals JSONL、performance.csv、trades.csv；dashboard 直接读取这些结果。

Operational Flow

项目真正是怎么跑起来的

第 1 步：启动 用户执行 ./start_local.sh，脚本会启动 elite_flow 和本地 dashboard，并把 PID 写到 engine/control/。
第 2 步：策略加载 main.py 检查比赛策略定义；如果 base_profile = custom，就直接加载自定义模块，比如 competition/strategies/elite_flow.py。
第 3 步：市场输入 标准引擎拿历史 OHLCV；elite_flow 同时订阅 orderbook、trades、1m OHLCV，再轮询 funding / open interest / long-short ratio。
第 4 步：信号与状态 标准引擎生成目标权重；elite_flow 则计算 flow_score、crowd_score、regime_score，合成 conviction，驱动状态机。
第 5 步：执行与记账 下单通过 OKX CLI，内部组合或自定义策略再把成交、杠杆、PnL、持仓信息写回日志与状态文件。
第 6 步：可视化 dashboard 读取 summary.json、elite_flow.log、bill / position 数据源，生成 NAV、持仓、决策流水和信号快照。
Data Elements

数据元总表：每个数据字段是什么，被用在哪里

你要求的“数据元”可以理解为：系统里真正被读取、计算、传递、记录和展示的字段。下面按来源分类，从输入到衍生到输出讲清楚。

A. 历史行情数据

OHLCV

字段：open、high、low、close、volume、funding_rate

来源：data/fetcher.py，优先 ccxt/OKX，spot 失败时可回退 yfinance。

用处：趋势、突破、横截面动量、carry、ATR 止损、相关性与回测。

B. 动态标的池数据

Universe

字段：symbol、24h volume、bid、ask、active 状态

来源：fetch_tradable_futures_symbols() 或 YOLO 路径里的 universe 发现逻辑。

用处：过滤稳定币、指数币、TradFi perp、低流动性品种，控制策略可交易范围。

C. 实时价格缓存

Latest Ticker Price

字段：symbol → latest price

来源：data/feed_ws.py 的 WebSocketPriceCache。

用处：60 秒风险检查、summary snapshot、当前 NAV 与持仓浮盈亏计算。

D. 盘口数据

Top-N Order Book

字段：bids[i][px, sz]、asks[i][px, sz]

来源：elite_flow WebSocket 深度流。

用处：计算多层 OFI，也就是订单流失衡，衡量买卖盘主动推进方向。

E. 实时成交流

Taker Buys / Sells

字段：成交方向、成交量、时间戳

来源：elite_flow trade stream。

用处：辅助判断主动买盘和主动卖盘强弱，配合 OFI 提高流向判断质量。

F. 拥挤度数据

Crowding Inputs

字段：open_interest、funding_rate、mark_price、index_price、long_short_ratio

来源：elite_flow 的 REST 轮询。

用处：判断多头过拥挤还是空头过拥挤，用于 squeeze / flush 检测。

G. Regime 输入

1m Close Series

字段：最近 60 个一分钟 close

来源：elite_flow 的 1m OHLCV 流。

用处：计算短期动量与波动率分位，判断当前是否处于可交易的趋势环境。

H. 派生信号

Signal Scores

字段：flow_score、crowd_score、regime_score、conviction

来源：策略内部计算。

用处：决定是否开仓、开多还是开空、加多少杠杆、是轻仓还是满仓。

I. 组合账本

Portfolio State

字段：cash、positions、realized_pnl_total、total_fees、peak_nav、drawdown_pct

来源：Portfolio 内部记账。

用处：风控、绩效统计、状态恢复、dashboard 展示。

J. 持仓级数据

Position Fields

字段：qty、avg_entry_price、mark、notional、weight、upnl、side

来源：Portfolio.snapshot() 生成。

用处：状态展示、风险检查、dashboard 持仓表。

K. 风险状态

Risk State

字段：circuit_breaker_state、vol_regime、risk_scalar、atr_stops

来源：core/risk.py 与 trading_engine.py。

用处：在调仓或 60 秒检查里缩仓、清仓、止损。

L. 日志与展示数据

Summary / Event Logs

字段：summary.json、每组合 .jsonl、performance.csv、trades.csv、decision_journal

来源：StructuredLogger 和自定义策略日志。

用处：status 命令、dashboard、赛后分析。

Strategies

策略大模块：现在有哪些策略，各自由什么组成

仓库里的策略分成两类。第一类是标准化组合策略，会接入通用引擎。第二类是比赛专用自定义策略，会自己维护事件循环和持仓状态。

标准策略

Combined Portfolio

组成：Trend Momentum + Cross-Sectional Momentum + Funding Carry。

含义：它不是一个单信号，而是三种 alpha sleeve 的组合。

怎么变化：daily / hourly profile 会改变 sleeve 权重、调仓节奏和风控阈值。

标准策略

Trend Momentum

组成：DMA crossover、breakout、time-series momentum、inverse-vol sizing。

含义：同一个币自己和自己的历史比，判断趋势方向。

怎么看变化：从负变正，说明趋势开始偏多；绝对值变大，说明趋势更强。

标准策略

Cross-Sectional Momentum

组成：多 lookback 收益率排名，做多最强、做空最弱。

含义：不是看某个币绝对涨跌，而是看“谁比别人更强”。

怎么看变化：排名变化越大，说明风格切换越明显。

标准策略

Funding Carry

组成：funding 均值平滑 + trend veto。

含义：利用永续合约 funding 偏斜来赚 carry，但会用趋势 veto 防止逆着趋势硬做。

怎么看变化：funding 越极端，信号越强；如果趋势反着来，信号会被 veto。

比赛策略

Elite Flow

组成：Multi-level OFI 45% + Crowding 35% + Regime Gate 20%。

含义：这是一个盘口驱动的挤仓/踩踏捕捉器，不是日频轮动模型。

怎么看变化：conviction 越高，说明方向、拥挤与环境越一致；状态机会从观望转为入场或持有。

比赛策略

YOLO Momentum

组成：多周期动量 + 高杠杆 + 分轮补仓 + ROI 目标停止。

含义：偏激进实验型策略，目标明确，风险也更集中。

怎么看变化：round 越往后，说明前面回合失败或止损，仓位风险在上升。

比赛编排策略

YOLO Orchestrator

组成：10 个顺序部署的 YOLO slot。

含义：不是新 alpha，而是把 YOLO 包装成一个更容易做 demo 验证的实验框架。

怎么看变化：看 slot 的 running / succeeded / drained 分布，就能知道整体实验健康度。

标准策略的输出形式：
price_data -> Insight(symbol, direction, weight, confidence)
          -> target_weights
          -> risk-adjusted weights

Elite Flow 的输出形式：
orderbook / trades / crowding / 1m close
  -> flow_score / crowd_score / regime_score
  -> conviction
  -> state machine
  -> order / leverage / exit decision
YOLO Routes

YOLO 的两条路线：单策略执行器与多槽位编排器

你提到的 YOLO 其实不是一条路线，而是两层结构。yolo_momentum 负责“怎么交易”，yolo_orchestrator 负责“怎么批量部署和管理多个 YOLO 实验槽位”。

路线 1

yolo_momentum

它是什么：一个高杠杆、单方向、单合约优先的激进动量执行器。

目标：用分轮保证金和高杠杆，追求累计投入资金的 20% ROI 目标。

关键机制：

  • 动态发现可交易 USDT-SWAP 标的，并过滤稳定币、股票 perp、过宽价差。
  • 根据 EMA、RSI、量能等做多周期动量判断。
  • 使用 round_margins = [50, 100, 200, 400] 的分轮投入。
  • 用高杠杆扩大收益，也允许在 liquidation 后进入下一轮。
  • 达到目标 ROI 后自动停机。
路线 2

yolo_orchestrator

它是什么：一个实验编排器，不直接生成 alpha。

目标：把 10 个 YOLO slot 依次部署出来，做更长时间的 demo 验证。

关键机制：

  • 每个 slot 有独立的预算、ROI、交易历史和状态。
  • 默认一次只允许 1 个 slot 真正活跃，避免共享账户下净持仓合并。
  • 一个 slot 结束后，立刻或按 stagger 时间部署下一个。
  • 把总实验结果写入 engine/logs/yolo_orchestrator.json。
YOLO 两条路线的关系：

yolo_momentum
  = 单个激进策略实例
  = 负责选标的、开仓、止损、追踪 ROI、轮次切换

yolo_orchestrator
  = 多个 yolo_momentum 的上层调度器
  = 负责 slot 排队、串行部署、整体 ROI 汇总、实验状态总览
YOLO Momentum 的核心数据元

round、cumulative_invested、target_roi_pct、default_lever、high_vol_lever、low_vol_lever、hard_stop_pct、cooldown_min

这些字段共同定义了“这一轮打多大、打多久、输了怎么切到下一轮”。

Orchestrator 的核心数据元

slot.status、started_at、finished_at、overall_pnl、overall_roi_pct、total_running、total_pending

这些字段不决定单笔交易，而是决定整个实验编排是否还健康。

为什么要单独讲两条路线

因为 yolo_momentum 是策略，yolo_orchestrator 是调度器。把两者混为一个东西，会误判修改点，也会误判日志与状态文件的职责。

YOLO 路线怎么看“变化代表什么”

字段或现象 变化时代表什么 你该怎么理解
round 上升 策略进入下一轮资金投入 说明上一轮没有完成目标，风险和资本占用在上升
roi_pct 接近 20% 接近 slot 成功线 这是 YOLO 的主要停机目标，不是普通净值指标而已
status = drained 预算轮次已经耗尽 该 slot 被视为失败实验，需要交给下一个 slot
status = succeeded 单 slot 达成目标 ROI 说明这一轮实验按设计完成
overall_roi_pct 上升 整个 orchestrator 实验变好 这是看 10 个 slot 合并表现的总指标
total_pending 下降 更多 slot 已部署 说明实验在往后推进，而不代表策略本身变强

YOLO Momentum：开仓规则到底是什么

YOLO 这一条线最容易被误解的点是：它有“3 个 gate”，但这 3 个 gate 讲的是入场条件，不是止损条件。也就是说，YOLO 不是先开了再想办法解释，而是先要求趋势、动量、量能同时对齐，才允许下单。

Gate 1: Trend Gate

看什么：15m、1H、4H 的 EMA 排列，再加 1H 相对 EMA50 的位置，最后做多数投票。

代码口径：ema_alignment >= 60%。

为什么重要：它先回答“现在大方向是不是顺着这笔交易”。如果大方向都不一致，后面的动量和量能很可能只是噪声。

Gate 2: Momentum Burst Gate

看什么：15m RSI 是否处在可追击区间，同时 MACD histogram 不能明显反着来。

多头口径：RSI 53-78；空头口径：RSI 22-47。

为什么重要：它确保“这不是已经衰竭的趋势”，而是还有一段惯性能继续跑。

Gate 3: Volume Gate

看什么：当前 15m 成交量是否明显高于最近平均量。

代码口径：volume_ratio >= 1.1。

为什么重要：YOLO 用高杠杆，最怕“方向判断对了，但没成交量跟随”，那种走势最容易抽回来。

YOLO Momentum：不是先过 gate，而是先打分、再选最强、再过 gate

真正的执行顺序不是“看到一个币就立刻过 gate”，而是先做一轮全市场横向比较。也就是说，YOLO 先对可交易 universe 里的所有合约逐个评分，挑出分数最高的那个，再让这个“第一名”去过 3 个 gate。只有第一名通过 gate，才允许真正开仓。

真实决策流水：

动态 universe
  -> 对每个合约抓 15m / 1H / 4H candles
  -> 计算 ADX / EMA alignment / volume ratio / funding / ATR% / RSI / MACD
  -> 算 composite score
  -> 选 score 最高的那个
  -> 让这个第一名过 Trend / Momentum / Volume 3 个 gate
  -> 通过后再决定杠杆、张数、止损单
  -> 下单
第一步

YOLO 怎么评分

YOLO 不是直接看 RSI 一个指标，而是先做一个 0 到 100 的综合评分。这个分数来自 5 块：

  • ADX 35%：趋势强不强。
  • EMA Alignment 25%：多周期 EMA 是否朝同一方向排列。
  • Volume 20%：当前量能是不是放大。
  • Funding Edge 10%：如果要做多，负 funding 更加分；如果要做空，正 funding 更加分。
  • ATR Sweet Spot 10%：波动率是否落在策略喜欢的区间。
第二步

为什么先评分再过 gate

因为 gate 解决的是“这个币现在能不能打”，评分解决的是“今天这么多币里，谁最值得先看”。

所以实际是：

  • 评分负责做横向排序。
  • gate 负责做最后的入场确认。
  • 如果第一名被 gate 挡掉，系统不会立刻开第二名，而是等下一轮重新扫描。
可以把 YOLO 理解成两层筛选： 第一层是“全市场谁最强”，第二层是“这个最强者现在是否真能打”。

YOLO Momentum：评分公式到底在看什么

评分块 权重 看什么 作用
ADX Score 35% 1H ADX 强度 回答“这是不是一个有趋势的市场”
Alignment Score 25% 15m / 1H / 4H EMA 方向投票 回答“多周期方向是否一致”
Volume Score 20% 当前量 / 平均量 回答“有没有量能确认”
Funding Score 10% funding 是否对这笔方向有利 回答“拥挤度是否给这笔交易加分”
ATR Sweet Spot 10% ATR% 是否落在策略喜欢的波动区间 回答“这只币的波动既不能太死，也不能乱飞”

这就是为什么你在看板上会看到一个币分数很高，但最后还是不开。高分只说明它在今天的全市场里很强，不代表它已经通过了最终入场确认。

用一句话概括 YOLO 的开仓逻辑： 先给全市场打分，选出第一名；再让第一名去过 3 个 gate；过了之后再根据 ATR 波动率先算目标杠杆，再被交易所最大杠杆和账户最大可下张数二次裁剪，最后才真正下单。

YOLO Momentum：杠杆是怎么决定的

YOLO 不是固定一个杠杆写死到底，而是先看波动率，再看交易所限制。

阶段 代码口径 含义
先算目标杠杆 ATR% > 3 -> 30x
ATR% < 1 -> 75x
其他默认 50x		 波动很大就降杠杆，波动很小才允许更高杠杆
再看交易所上限 如果目标杠杆大于该合约允许最大杠杆，就自动降到该合约的 max leverage 比如想开 50x，但这个币只允许 20x，就会变成 20x
最后看账户最大可下张数 如果按当前保证金和杠杆算出来的张数大于账户可下上限，就自动缩小 这一步解决“理论上能开，但账户实际下不了那么多张”的问题

YOLO Momentum：什么时候真正开仓

YOLO 不是扫到高分就立刻下单。它真正下单要同时满足下面 6 个条件：

条件 1：这一轮还在 HUNTING 只有当前状态是 HUNTING，系统才会继续扫描。
条件 2：全市场评分已经选出第一名 先完成一轮 universe 扫描，拿到 selected_candidate。
条件 3：第一名通过 3 个 gate 趋势、动量、量能三关必须同时过线。
条件 4：杠杆和张数在交易所允许范围内 目标杠杆会被交易所上限裁剪，张数会被 account max-size 裁剪。
条件 5：这一轮风险桶资金够 现在 live 版本已经改成逐仓，还会检查这一轮的 margin + reserve 资金桶是否够，不够就进 RECHARGE_REQUIRED。
条件 6：下单成功 真正成功标准不是“发出信号”，而是 kit 下单成功，并进入 IN_POSITION。

YOLO Momentum：止损和退出规则是什么

退出规则 触发条件 系统会做什么
Target Hit 浮盈达到当前轮次目标利润 立即平仓，状态记为成功，本 slot 停机
Hard Stop 浮亏达到当前轮 current_margin 的 60% 平仓，记为 ROUND_LOST，进入冷却并准备下一轮
Trailing Stop 先达到目标的 50%，之后利润从峰值明显回撤 平仓，锁住一部分利润，不一定结束整个 slot
Reversal Detection 综合反转分数 >= 0.45 立即平仓；如果只是 >= 0.30，则先收紧止损
Time Decay 持仓超过 4 小时，但利润还不到目标的 5% 认为动量已经钝化，主动退出
Liquidation Detection 当亏损较深时，交易所上已经查不到该持仓 视作本轮失败，进入下一轮资金分配逻辑

YOLO Momentum：什么时候关仓

一旦进入 IN_POSITION，系统每隔一段时间会做一次 reconcile。它不是盯着一个指标，而是按一套优先级顺序检查：

第 1 优先级：Target Hit 如果浮盈已经达到当前轮目标利润，直接平仓，这是 YOLO 最理想的退出方式。
第 2 优先级：Hard Stop 如果浮亏达到本轮 current_margin × hard_stop_pct，直接退出。现在代码里还会在下单时同步挂一个硬止损单，防止只靠本地轮询来平仓。
第 3 优先级：Trailing Stop 如果盈利先达到目标的一半，系统会启动 trailing；之后如果利润从峰值回撤到阈值以下，就锁盈离场。
第 4 优先级：Time Decay 如果已经抱了 4 小时，但利润还是低于目标的 5%，说明动量钝化，主动退出。
第 5 优先级：Reversal Detection 系统还会额外算一个反转分数，由 volume climax、RSI divergence、EMA cross、funding extreme、MACD shrinking 组成。分数够高就直接平；分数中等则先收紧止损。
第 6 优先级：仓位消失检测 如果亏损已经很深，同时交易所上查不到仓位，系统会把它视作本轮失败，进入下一轮逻辑。

如果第一轮就打到硬止损，YOLO 不会继续摊平原仓位。它会先冷却，再把轮次切到下一轮，把保证金从 50 → 100 → 200 → 400 逐级提高，然后重新跑一次全市场选标和 3 个 gate，目标是用下一轮交易把前面的亏损和新的 20% ROI 目标一起追回来。

Elite Flow

当前主策略 `elite_flow` 详细说明

README 明确说明了当前最稳定可运行的路径是 elite_flow。它是这个仓库里最值得优先理解的策略，因为 dashboard、start script 和日志解析都围绕它做了增强。

信号结构

三段式信号

  • Flow：看盘口与主动成交，判断现在的短线力量到底往哪边推。
  • Crowding：看 OI、funding、premium、L/S ratio，判断对手盘是不是已经站得太挤。
  • Regime：看短期动量和波动率分位，避免在震荡或极端异常波动里乱打。
配置重点

阈值与权重

  • flow_weight = 0.45
  • crowd_weight = 0.35
  • regime_weight = 0.20
  • entry_threshold = 0.40
  • full_threshold = 0.60
  • max_positions = 1

Flow 变大说明什么

盘口深度在你想要的方向上被持续吃穿，或者该方向不断补单强化。它代表“眼前这一分钟，主动推动价格的是谁”。

Crowd 变大说明什么

市场一边倒地拥挤。如果价格一旦突破，对拥挤一侧会形成挤压。这个分数的重点不是顺势，而是“对手盘会不会被迫踩踏”。

Regime 变差说明什么

要么太震荡，没有方向；要么太极端，容易失控。Regime Gate 的任务是过滤掉“信号看着有，但环境不支持”的时刻。

新手最容易误解的一点： elite_flow 不是单纯“趋势策略”。它本质上是在寻找“方向 + 拥挤 + 环境”同时对齐的短线机会。

Elite Flow：开仓规则怎么理解

elite_flow 没有像 YOLO 那样把规则写成“Trend / Momentum / Volume 三个 gate”，但它本质上也有三层入场门，只是表达方式是“三个分数组合成 conviction”。你可以把它理解成一个更交易员化的三段式开仓系统。

Gate 1: Flow

看什么：盘口 top-N 深度失衡 + 主动成交方向。

代码口径：先算 flow_score；如果 abs(flow_score) 太小，直接不做。

意义：先确认“眼前这一刻是谁在真正推动价格”，这是 Elite 最重要的前置条件。

Gate 2: Crowding

看什么：OI、funding、premium、long/short ratio 是否说明某一边已经太拥挤。

代码口径：形成 crowd_score，只在 squeeze / flush 结构清晰时给分。

意义：不是单纯顺势，而是判断“如果价格再往这边走，对手盘会不会被迫补仓”。

Gate 3: Regime

看什么：短周期趋势和波动率环境是不是可交易。

代码口径：形成 regime_score；如果趋势和 flow 打架，会强行降权。

意义：防止在震荡、极端波动或错误环境里把微观信号过度解释。

最终它不是简单“3 个都过就开”，而是走这条链路：

flow_score / crowd_score / regime_score
  -> 加权合成 conviction
  -> 先检查 flow 是否足够强
  -> 再检查 flow 和 regime 是否同向
  -> conviction 达到 entry_threshold 才允许入场
  -> conviction 更高时，从 PROBE 进入 FULL

所以 Elite 的开仓规则更像：“先看微观流是否强，再看拥挤结构是否支持，再看环境是否允许，最后把三者压缩成一个 conviction，再决定是不开、试探仓还是满仓。”

Elite Flow：止损和退出规则是什么

退出规则 触发条件 系统会做什么
Flow Reversal 持仓方向和 flow_score 明显反着来，而且反向流已经很强 立即退出，因为 Elite 把微观流当成第一优先级
Stop Loss pnl_pct <= -2.0% 硬止损，平掉当前仓位
Take Profit pnl_pct >= 3.5% 且 flow 已经衰减 止盈退出，不追求无限抱单
Max Hold 持仓时间超过 max_hold_min 无论盈亏都退出，避免短线策略拖成长线赌单
Daily Loss Stop 当日累计亏损超过 5% 停止开新仓，让系统进入保护状态

跟 YOLO 相比，Elite 的风险控制更“硬”和更交易员化：它不靠多轮补仓去追回，而是更强调单笔快速验证、错了马上砍、对了在流衰减时离场。

Core Trading System

核心交易系统：非常详细的拆解

这一部分不是讲策略思想，而是讲系统是如何把想法变成交易的。你如果以后要维护这个仓库，最该读懂的是这里。

核心对象

1. `TradingEngine` 是调度中枢

它是什么：一个长期运行的 orchestrator。

它解决什么问题：让多个组合自动运行、自动调仓、自动风控、自动保存状态。

它怎么工作：

  • start() 根据配置创建或恢复多个 Portfolio。
  • 启动 WebSocketPriceCache，降低风险检查时的 REST 成本。
  • 写 PID 文件，写启动 summary。
  • run_loop() 每轮检查是否需要 rebalance、是否需要 risk check。
  • 每个循环结束后写 summary.json 和 engine_state.json。

为什么这样设计：这让“策略逻辑”和“运行维护逻辑”分开，后续出问题时更容易定位是在信号层、风控层还是进程层。

核心对象

2. `TradingAlgorithm` 是调仓流水线

它是什么：无状态的调仓 pipeline。

它为什么重要：项目里最通用、最可复用的抽象都在这里。

单次 rebalance 内部发生什么：

  • 从 price_data 取当前 close，估算调仓前 NAV。
  • 调用 alpha.generate() 输出 raw insights。
  • 调用 portfolio_construction.create_targets() 输出目标权重。
  • 调用 risk.adjust() 进一步缩放或清零权重。
  • 调用 execution.execute() 把目标权重变成真实 trade。
  • 更新组合快照、NAV 历史、peak NAV，并写结构化日志。

这套设计非常像 LEAN 的模块化模型，但代码更贴近这个项目自己的比赛和部署场景。

核心对象

3. Alpha 层

标准路径下由 CombinedAlphaModel 统一调度多个 sleeve。它把策略输出标准化成 Insight 列表，每条 insight 至少包含 symbol、方向、权重、置信度和来源。

为什么重要：只有先把观点表达成统一对象，后面的组合构建和风控才能完全复用。

核心对象

4. Portfolio Construction 层

由 SignalFilteredPortfolioModel 做三件事：阈值过滤、集中度过滤、杠杆和净敞口约束。

为什么重要：策略可能会产生很多小而杂的信号，但真正能下场交易的通常只应该留下最强的一部分。

核心对象

5. Risk 层

用 CompositeRiskModel 串联多个模型，至少包括回撤断路器、波动率 regime、相关性 watchdog。

为什么重要：它不是“止损补丁”，而是决定系统还能不能继续开仓的控制塔。

核心对象

6. Execution 层

标准路径里会把目标权重转成交易动作，再调用 Broker。真实下单不走 ccxt，而是走 OKX CLI，这样能满足比赛对标签和路径的要求。

关键点：读和写是分离的。读市场用 ccxt，写订单用 CLI。

核心对象

7. Portfolio 账本层

项目内部净值不是直接复制交易所余额，而是自己用 record_trade() 算已实现收益、手续费、持仓成本和现金。

关键点：这让多策略共享一个账户时，仍能保持各自独立的表现归因。

核心对象

8. Logger 与 Dashboard

StructuredLogger 把系统状态从“运行中的对象”变成“可消费的文件格式”。dashboard 不需要直接 import 引擎内部对象，只读文件即可。

关键点：这是系统工程里非常稳的一种解耦方式。

核心交易系统的完整链路

1. main.py 接收命令
2. TradingEngine.start() 创建或恢复 Portfolio
3. WebSocketPriceCache 启动，准备轻量实时价格
4. run_loop() 进入长期循环
5. 到了调仓时间：
   5.1 拉取 price_data
   5.2 TradingAlgorithm.rebalance()
       - Alpha 生成 insight
       - PortfolioConstruction 生成目标权重
       - Risk 做断路器 / 波动 / 相关性调整
       - Execution 生成订单
       - Portfolio 记账
       - StructuredLogger 写日志
   5.3 重新计算 ATR stops
6. 每 60 秒：
   6.1 读取实时价格
   6.2 计算当前 NAV 与 drawdown
   6.3 必要时 emergency close 或 ATR stop
7. 每轮循环后：
   7.1 写 summary.json
   7.2 持久化 engine_state.json
8. dashboard / status 命令读取这些文件展示结果

风险系统为什么值得重点看

风控模型 做什么 什么时候触发 动作
Drawdown Circuit Breaker 跟踪从峰值 NAV 的回撤 回撤超过 reduced 或 cash 阈值 缩仓 50% 或直接归零
Vol Regime 根据 NAV 历史波动决定风险倍数 高波动或低波动 regime 统一乘一个风险 scalar
Correlation Watchdog 看整个 universe 是否都一起动 平均相关性过高 所有仓位乘 0.6
ATR Stops 给单个仓位设置动态止损线 当前价格击穿 stop_price 单仓平掉

自定义策略路径里的核心交易系统

Elite Flow 系统结构
  • 维护每个 symbol 的 SymbolState。
  • 在 state 里记录 OFI 计算器、regime gate、crowding model、最近成交时间和最近价格。
  • 不断更新 flow_score、crowd_score、regime_score。
  • 把三个分数合成为 conviction，再驱动状态机。
为什么它单独实现
  • 因为它依赖盘口 top-5 深度，不是简单 OHLCV 因子。
  • 因为它的调度频率比标准日频/4 小时调仓快很多。
  • 因为它要自己管理杠杆、最小持有时间、逐笔 reconcile 和 entry / exit 事件日志。
  • 因为它更像一个微观结构事件驱动策略，而不是传统组合轮动策略。
YOLO Momentum 系统结构
  • 先做动态 universe 发现，再挑流动性和价差合格的合约。
  • 用 EMA、RSI、量能等组合成单标的动量判断。
  • 根据波动率在 30x、50x、75x 之间调整杠杆。
  • 用轮次资金和 trailing / hard stop / time decay 控制退出。
YOLO Orchestrator 系统结构
  • 把一个高风险策略包成多个 slot 的实验流程。
  • 每个 slot 单独跟踪投入、PnL、ROI、当前仓位和交易记录。
  • 通过 stagger 机制错峰部署，避免多个 slot 同时打架。
  • 把单策略层的问题，提升成“实验管理”和“资金编排”问题。
最重要的工程结论： 这个仓库的核心交易系统不是某一个类，而是“调度 + 数据 + 决策 + 执行 + 记账 + 日志”六件事拼起来的整体。
